# 유틸리티 타입

> TL;DR: 기존 타입을 입력받아 변형된 새 타입을 반환하는 제네릭 타입. 매핑된 타입(mapped type)과 조건부 타입으로 구현돼 있어 타입 중복 정의를 제거한다.

## What

유틸리티 타입은 TypeScript가 전역으로 제공하는 제네릭 타입이다. 타입 인자로 기존 타입을 받아 키를 선택·제거·변형한 새 타입을 만든다. 대부분 매핑된 타입 `{ [K in keyof T]: ... }` 문법으로 정의돼 있다.

자주 쓰는 것들:

- `Partial<T>`: T의 모든 속성을 선택적(`?`)으로 변환.
- `Required<T>`: T의 모든 속성을 필수로 변환(`?` 제거).
- `Readonly<T>`: T의 모든 속성에 `readonly` 부착.
- `Pick<T, K>`: T에서 키 `K`(유니온)만 골라낸 타입.
- `Omit<T, K>`: T에서 키 `K`를 제외한 타입.
- `Record<K, V>`: 키 `K`, 값 타입 `V`인 객체 타입.
- `ReturnType<F>`: 함수 타입 F의 반환 타입을 추출.

## Why it matters

API 응답 타입 하나에서 폼 입력 타입, 수정 요청 타입, 화면 표시 타입을 파생시킬 때 쓴다. 원본 `User` 타입을 정의하고 `Omit<User, "id" | "createdAt">`로 생성 요청 타입을, `Partial<User>`로 수정 요청 타입을 만들면 필드 추가 시 한 곳만 고쳐도 파생 타입이 함께 따라온다.

면접에서는 "Pick과 Omit 차이", "Partial을 매핑된 타입으로 직접 구현해보라", "Record와 인덱스 시그니처 차이" 같은 형태로 나온다.

## How

기존 타입에서 파생 타입 만들기:

```ts
interface User {
  id: number;
  name: string;
  email: string;
  createdAt: Date;
}

// 생성 요청: 서버가 만드는 id, createdAt 제외
type CreateUserDto = Omit<User, "id" | "createdAt">;
// { name: string; email: string; }

// 수정 요청: 일부 필드만 보냄
type UpdateUserDto = Partial<Omit<User, "id">>;
// { name?: string; email?: string; createdAt?: Date; }

// 목록 화면: 일부 필드만 표시
type UserSummary = Pick<User, "id" | "name">;
// { id: number; name: string; }
```

Record로 매핑 객체 만들기:

```ts
type Role = "admin" | "user" | "guest";

const permissions: Record<Role, string[]> = {
  admin: ["read", "write", "delete"],
  user: ["read", "write"],
  guest: ["read"],
};
// Role 유니온에 멤버를 추가하면 permissions에서 누락 키를 컴파일 에러로 잡는다.
```

ReturnType으로 함수 반환 타입 추출:

```ts
function createStore() {
  return { count: 0, increment() {} };
}

type Store = ReturnType<typeof createStore>;
// { count: number; increment: () => void; }
```

Partial을 매핑된 타입으로 직접 구현(표준 라이브러리 정의와 동일):

```ts
type MyPartial<T> = {
  [K in keyof T]?: T[K];
};

type MyPick<T, K extends keyof T> = {
  [P in K]: T[P];
};

// Omit은 Pick + Exclude 조합으로 구현된다.
// 표준 라이브러리는 K를 keyof any로 제약해, T에 없는 키를 넣어도 에러가 안 난다(아래 Pitfalls 참고).
type MyOmit<T, K extends keyof any> = MyPick<T, Exclude<keyof T, K>>;
```

`keyof T`로 키 유니온을 얻고, `[K in ...]`로 각 키를 순회하며 `?`나 `readonly` 같은 수정자를 붙이거나 떼는 방식이다.

## Pitfalls

`Omit`은 두 번째 인자 키가 T에 없어도 에러를 내지 않는다. 오타가 조용히 통과한다.

```ts
interface User { id: number; name: string; }

type A = Omit<User, "nmae">; // 에러 없음, "nmae"는 무시되고 User 그대로
// Pick은 다르다: 키가 없으면 컴파일 에러
type B = Pick<User, "nmae">; // 에러: "nmae"는 User의 키가 아님
```

`Partial`은 한 단계(shallow)만 적용된다. 중첩 객체 속성은 여전히 필수로 남는다.

```ts
interface Config {
  server: { host: string; port: number };
}

type P = Partial<Config>;
const c: P = { server: { host: "localhost" } };
// 에러: server 안의 port가 빠짐. server는 선택이지만 그 내부는 그대로 필수
```

`Record<string, V>`로 만든 객체에 임의 키로 접근하면 값이 `V`로 좁혀지지만 런타임에는 `undefined`일 수 있다. `noUncheckedIndexedAccess` 옵션을 켜지 않으면 타입이 실제와 어긋난다.

## 직접 확인

1. `MyRequired<T>` 와 `MyReadonly<T>` 를 매핑된 타입으로 직접 작성하라. `Required`는 `-?` 수정자로 `?`를 제거하고, `Readonly`는 `readonly`를 붙인다. 표준 타입과 결과가 일치하는지 동일 인터페이스에 둘 다 적용해 호버로 비교하라.
2. 본인 코드의 인터페이스 하나를 골라 `Pick`/`Omit`/`Partial`로 파생 타입 3개를 만들어보고, 원본에 필드를 1개 추가했을 때 파생 타입들이 어떻게 바뀌는지 호버로 확인하라.

## 스스로 답하기

- `Pick<T, K>`와 `Omit<T, K>`는 존재하지 않는 키를 줬을 때 동작이 어떻게 다르고, 그 차이가 왜 생기는가?
- `Partial<T>`가 중첩 객체에는 적용되지 않는 이유는 무엇이고, 깊은 Partial이 필요하면 어떻게 구현하는가?
- `Record<K, V>`와 인덱스 시그니처 `{ [key: string]: V }`는 키 제약 측면에서 어떻게 다른가?

## 내 프로젝트 연결

컴포넌트 props 타입을 다른 곳에서 통째로 다시 선언하고 있다면, 원본 데이터 타입에서 `Pick`으로 필요한 필드만 골라 props를 정의해보라. 예: `type CardProps = Pick<User, "name" | "email">`. 폼 상태를 다룰 때 모든 필드가 처음에 비어 있다면 `Partial<FormValues>`로 초기 상태 타입을 잡고, 제출 직전에 필수 검증으로 좁힌다. 같은 필드 묶음을 여러 인터페이스에 중복 작성한 부분을 찾아 하나의 원본 타입 + 유틸리티 타입 파생으로 줄여보라.
