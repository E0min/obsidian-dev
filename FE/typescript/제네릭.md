# 제네릭

> TL;DR: 타입을 `<T>` 같은 매개변수로 받아 호출 시점에 결정하는 기능. 입력 타입과 출력 타입을 하나의 변수로 묶어, `any`로 잃어버리는 타입 정보를 보존한다.

## What

타입을 값처럼 매개변수로 받는 기능. 함수가 인자를 받듯, 타입 매개변수 `<T>`를 선언하면 호출/사용 시점에 실제 타입이 채워진다.

```ts
function identity<T>(value: T): T {
  return value;
}

const a = identity<string>("hi"); // a: string
const b = identity(42);           // b: number (T를 number로 추론)
```

동작 순서:
1. `identity<T>`에서 `T`는 아직 미정인 타입 변수.
2. 호출 시 인자 타입을 보고 `T`를 추론하거나, `<string>`처럼 명시한다.
3. 반환 타입 `T`가 입력 타입과 같은 값으로 채워져 입력·출력이 연결된다.

인터페이스와 타입 별칭도 같은 방식으로 매개변수를 받는다.

```ts
interface Box<T> {
  value: T;
}

const numBox: Box<number> = { value: 1 };
```

## Why it matters

FE 실무에서 API 응답을 다룰 때 가장 자주 부딪힌다. `fetch` 래퍼나 `axios`, `tanstack-query`의 반환 타입을 `any`로 두면 이후 모든 접근에서 자동완성과 타입 검사가 사라진다. 제네릭으로 응답 타입을 받으면 호출하는 쪽마다 다른 데이터 형태를 그대로 유지하면서 검사를 받는다. 면접에서는 "`any`와 제네릭의 차이", "제네릭 제약(`extends`)을 언제 쓰는가"를 자주 묻는다.

## How

### 1. any와의 차이

```ts
function firstAny(arr: any[]): any {
  return arr[0];
}
const x = firstAny(["a", "b"]); // x: any → x.toUpperCase()도, x.toFixed()도 통과

function first<T>(arr: T[]): T {
  return arr[0];
}
const y = first(["a", "b"]); // y: string → y.toFixed()는 컴파일 에러로 걸림
```

`any`는 검사를 끄지만, 제네릭은 입력 배열의 요소 타입을 `T`로 잡아 반환 타입까지 연결한다.

### 2. extends 제약

`T`에 아무 타입이나 들어오면 멤버 접근을 보장할 수 없다. `extends`로 "최소 이 형태는 만족해야 한다"는 조건을 건다.

```ts
function getLength<T extends { length: number }>(value: T): number {
  return value.length;
}

getLength("hello");        // 5
getLength([1, 2, 3]);      // 3
getLength({ length: 10 }); // 10
getLength(42);             // 에러: number에는 length가 없음
```

### 3. 기본 타입 매개변수

매개변수를 명시하지 않거나 추론이 안 될 때 쓸 기본값을 `= 타입`으로 지정한다.

```ts
interface ApiResponse<T = unknown> {
  data: T;
  status: number;
}

const r1: ApiResponse = { data: {}, status: 200 };          // T = unknown
const r2: ApiResponse<string[]> = { data: [], status: 200 }; // T = string[]
```

### 4. API 응답 래퍼 타입

서버 응답을 `{ data, message, code }` 형태로 감싸는 경우, 데이터부만 제네릭으로 받는다.

```ts
interface ApiResult<T> {
  data: T;
  message: string;
  code: number;
}

interface User {
  id: number;
  name: string;
}

async function fetchJson<T>(url: string): Promise<ApiResult<T>> {
  const res = await fetch(url);
  return res.json();
}

const result = await fetchJson<User>("/api/me");
result.data.name; // string으로 자동완성, data.age는 에러
```

호출하는 쪽이 `<User>`만 바꾸면 `data` 타입이 그대로 따라온다.

## Pitfalls

타입 매개변수를 한 번도 안 쓰거나 반환 위치에만 쓰면, 사실상 타입 단언(`as T`)과 같아 제네릭으로 둘 이유가 없다.

```ts
// 안티패턴: T가 반환에만 쓰여서 사실상 캐스팅
function parse<T>(json: string): T {
  return JSON.parse(json); // 런타임 형태를 검사하지 않음
}
const u = parse<User>("{}"); // u.name 접근은 통과하지만 런타임엔 undefined
```

`extends` 없이 멤버에 접근하면 막힌다.

```ts
function bad<T>(value: T) {
  return value.length; // 에러: T에 length가 있다는 보장 없음
}
```

기본 타입 매개변수를 `any`로 두면 제네릭의 이점이 사라진다. 채우지 못할 땐 `unknown`을 기본값으로 둔다.

## 직접 확인

1. `firstAny`와 `first<T>`를 에디터에 직접 입력하고, 반환값 뒤에 `.`을 찍어 자동완성 목록을 비교한다. `any` 쪽에서는 존재하지 않는 메서드도 제안되는지 확인한다.
2. `getLength<T extends { length: number }>`에 `getLength(42)`를 넣어 보고, 에러 메시지에 어떤 제약 위반이 표시되는지 그대로 읽어 본다. 그다음 `extends` 절을 지웠을 때 에러 위치가 어떻게 바뀌는지 비교한다.

## 스스로 답하기

1. `any`로 받은 값과 제네릭 `T`로 받은 값은 타입 검사에서 무엇이 다른가? 자동완성 측면에서 설명해 보라.
2. `function f<T>(x: T): T`에서 `T extends string`을 추가하면 사용처가 어떻게 제한되는가?
3. API 응답 래퍼 `ApiResult<T>`에서 `T`에 기본 타입 매개변수를 줄 때 `unknown`과 `any` 중 무엇을 골라야 하고 그 이유는?

## 내 프로젝트 연결

`tanstack-query`의 `useQuery<User>(...)`처럼 응답 타입을 제네릭으로 지정한 곳, 또는 직접 만든 `fetchJson<T>` 같은 API 응답 래퍼를 떠올려 본다. 본인 코드에서 응답을 `any`로 둔 함수 한 곳을 찾아 `<T>`로 바꾸고, 호출하는 쪽에서 실제 타입을 넣은 뒤 `data` 접근에 자동완성이 살아나는지 확인한다.
