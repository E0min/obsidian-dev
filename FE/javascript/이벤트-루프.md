# 이벤트 루프

> TL;DR: JS 엔진은 콜 스택이 빈 순간마다 마이크로태스크 큐를 전부 비우고, 그다음 매크로태스크 큐에서 하나만 꺼내 실행한다.

## What

JS 런타임은 단일 스레드로 콜 스택 하나를 쓴다. 이벤트 루프는 콜 스택이 비었는지 반복 확인하고, 비면 대기 중인 콜백을 큐에서 꺼내 스택에 올리는 반복 장치다.

큐는 우선순위가 다른 두 종류로 나뉜다.

- **마이크로태스크 큐**: `Promise.then`/`catch`/`finally` 콜백, `queueMicrotask`, `await` 이후 코드, `MutationObserver`.
- **매크로태스크 큐(태스크 큐)**: `setTimeout`, `setInterval`, `setImmediate`(Node), DOM 이벤트 콜백, I/O 콜백.

한 사이클의 처리 순서는 고정돼 있다. (1) 현재 동기 코드를 콜 스택에서 끝까지 실행한다. (2) 콜 스택이 비면 마이크로태스크 큐를 전부 비운다. 이때 새로 추가된 마이크로태스크도 같은 차례에 모두 처리한다. (3) 매크로태스크 큐에서 콜백을 딱 하나 꺼내 실행한다. (4) 다시 (2)로 돌아가 마이크로태스크 큐를 비운다.

## Why it matters

`setTimeout(fn, 0)`이 즉시 실행되지 않고 `Promise.then`보다 늦게 도는 이유가 이 우선순위에서 나온다. 면접에서 실행 순서 예측 문제로 자주 나온다.

실무에서는 상태 업데이트 타이밍, race condition, 무한 마이크로태스크로 인한 렌더 블로킹을 디버깅할 때 부딪힌다. React 18의 자동 배칭, `Promise` 체이닝 순서, `await`로 분기된 코드가 언제 다시 실행되는지를 추적하려면 이 순서를 알아야 한다.

## How

실행 순서 예측 예제다.

```js
console.log("1: 동기");

setTimeout(() => console.log("2: 매크로(setTimeout)"), 0);

Promise.resolve().then(() => console.log("3: 마이크로(then)"));

queueMicrotask(() => console.log("4: 마이크로(queueMicrotask)"));

console.log("5: 동기");

// 출력 순서: 1 → 5 → 3 → 4 → 2
// 동기 코드(1,5) 먼저, 마이크로태스크 전부(3,4), 매크로태스크 하나(2)
```

마이크로태스크가 매크로태스크보다 항상 먼저 비워지는 예제다.

```js
setTimeout(() => console.log("매크로 1"), 0);
setTimeout(() => console.log("매크로 2"), 0);

Promise.resolve()
  .then(() => console.log("마이크로 A"))
  .then(() => console.log("마이크로 B"));

// 출력: 마이크로 A → 마이크로 B → 매크로 1 → 매크로 2
// 매크로 1을 꺼내기 전에 마이크로 큐(A, B)를 전부 비운다.
```

`async/await`는 `Promise.then`으로 풀린다. `await` 다음 줄은 마이크로태스크로 예약된다.

```js
async function run() {
  console.log("1");
  await null; // 여기서 함수가 중단되고, 이후 코드는 마이크로태스크로 예약됨
  console.log("3"); // 마이크로태스크로 실행
}

console.log("0");
run();
console.log("2"); // run() 호출 직후의 동기 코드

// 출력: 0 → 1 → 2 → 3
```

## Pitfalls

마이크로태스크 안에서 마이크로태스크를 무한히 추가하면 매크로태스크와 렌더링이 영원히 막힌다.

```js
function loop() {
  Promise.resolve().then(loop); // 큐가 절대 비지 않음 → 브라우저 멈춤
}
loop();
// setTimeout으로 바꾸면 매크로태스크라 사이 사이 렌더가 들어간다.
```

`setTimeout(fn, 0)`의 0은 즉시가 아니라 최소 지연이다. 동기 코드와 마이크로태스크가 모두 끝난 뒤에야 실행되고, HTML 명세상 타이머 중첩이 5단계를 넘으면 브라우저가 최소 4ms를 강제한다.

```js
const arr = [0, 1, 2];
arr.forEach((i) => setTimeout(() => console.log(i), 0));
console.log("동기 끝");
// 출력: 동기 끝 → 0 → 1 → 2
// 콜백은 모두 매크로태스크라 동기 코드가 끝난 뒤에 실행된다.
```

## 직접 확인

1. 위 첫 번째 예제(`console.log("1")` ~ 출력 `1 → 5 → 3 → 4 → 2`)를 브라우저 콘솔에 직접 붙여 넣고, 실행하기 전에 종이에 출력 순서를 적어 본다. 그다음 `then` 콜백 안에 `setTimeout` 하나, `Promise.resolve().then` 하나를 더 중첩해 넣고 다시 예측한 뒤 실제 출력과 대조한다.
2. `performance.now()`를 동기 시작 지점, `queueMicrotask` 콜백 안, `setTimeout(fn, 0)` 콜백 안 세 군데에서 찍어 시간 차를 측정한다. 마이크로태스크와 매크로태스크 사이의 실제 지연이 몇 ms인지 기록한다.

## 스스로 답하기

1. `setTimeout(fn, 0)`이 0ms인데도 같은 틱의 `Promise.then`보다 늦게 실행되는 이유를 큐 두 종류와 이벤트 루프 사이클 순서로 설명할 수 있는가?
2. 마이크로태스크 큐 처리 중에 새 마이크로태스크가 추가되면 그것은 이번 차례에 처리되는가, 다음 매크로태스크 뒤로 밀리는가?
3. `async function` 안의 `await` 한 줄이 실행 순서에 어떤 변화를 만드는지, 그것이 `Promise.then` 등록과 어떻게 같은지 코드로 보일 수 있는가?

## 내 프로젝트 연결

chatGraph의 optimistic update에서 mutation 성공 콜백이 마이크로태스크로 처리되는 타이밍을 확인해 본다. `onSuccess` 콜백이 동기 setState 직후 같은 틱에 도는지, 다음 마이크로태스크 차례에 도는지 추적하면 캐시 갱신과 화면 반영 순서가 보인다.

Fit의 `setRemoteDescription` 직후 ICE candidate를 flush하는 코드에서, `await setRemoteDescription(...)` 다음 줄이 마이크로태스크로 예약되는 점을 떠올린다. candidate를 큐에 모아 두었다가 flush할 때 `await`와 `then`이 각각 언제 실행되는지 순서를 따라가 본다. 본인 코드의 `await`/`then`이 동기 코드와 `setTimeout` 사이 어디에 끼는지 직접 로그로 찍어 확인하라.
