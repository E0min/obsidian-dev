# 프로토타입 체인

> TL;DR: 모든 객체는 내부 슬롯 `[[Prototype]]`으로 다른 객체를 가리키고, 프로퍼티를 찾을 때 자신에게 없으면 이 링크를 타고 위로 올라가며 탐색한다. 이 연결 사슬이 프로토타입 체인이다.

## What

객체는 자신의 프로퍼티를 직접 갖거나, 다른 객체를 부모로 참조한다. 이 부모 참조가 내부 슬롯 `[[Prototype]]`이고, 표준 접근자는 `Object.getPrototypeOf(obj)`다. (`__proto__`는 같은 슬롯에 접근하는 레거시 게터/세터.)

프로퍼티 읽기 동작 순서:

1. 객체 자신에게 해당 키가 있으면 그 값을 반환한다.
2. 없으면 `[[Prototype]]`이 가리키는 객체에서 같은 키를 찾는다.
3. 거기에도 없으면 그 객체의 `[[Prototype]]`으로 다시 올라간다.
4. `null`(체인의 끝, 보통 `Object.prototype`의 `[[Prototype]]`)에 도달하면 `undefined`를 반환한다.

쓰기와 삭제는 체인을 타지 않는다. `obj.x = 1`은 항상 `obj` 자신에 프로퍼티를 만들거나 덮어쓴다.

## Why it matters

- 메서드를 인스턴스마다 복제하지 않고 공유하는 방식이다. 배열 1만 개를 만들어도 `.map`은 `Array.prototype`에 하나만 있고 모든 인스턴스가 체인으로 참조하므로, 메모리를 아끼고 메서드가 어디 정의됐는지 추적할 수 있다.
- 라이브러리 인스턴스(`d3.select(...)`, axios 인스턴스 등)의 메서드가 어디서 오는지, `instanceof`가 왜 그렇게 동작하는지 설명한다.
- 면접 단골이다. `prototype` 프로퍼티와 `__proto__`의 차이, `new`의 동작, `Object.create`를 묻는 질문은 모두 체인 위에 있다.

## How

### 1. 생성자 함수의 메서드 공유

```js
function Dog(name) {
  this.name = name;          // 인스턴스 자신의 프로퍼티
}
Dog.prototype.bark = function () {  // 공유 메서드는 prototype에
  return `${this.name}: 멍`;
};

const a = new Dog("초코");
const b = new Dog("바둑");

a.bark();                          // "초코: 멍"
a.bark === b.bark;                 // true (같은 함수를 공유)

// 체인 구조:
// a ──[[Prototype]]──▶ Dog.prototype ──[[Prototype]]──▶ Object.prototype ──▶ null
//  └ name:"초코"        └ bark, constructor              └ toString, hasOwnProperty
```

`new Dog()`가 하는 일을 풀면 이렇다.

```js
function myNew(Ctor, ...args) {
  const obj = Object.create(Ctor.prototype); // obj.[[Prototype]] = Ctor.prototype
  const ret = Ctor.apply(obj, args);         // this = obj 로 본문 실행
  return typeof ret === "object" && ret !== null ? ret : obj;
}
```

여기서 핵심: `Dog.prototype`은 "Dog로 만든 인스턴스의 부모가 될 객체"이고, 인스턴스의 `[[Prototype]]`(`a.__proto__`)이 바로 그 객체를 가리킨다. 즉 `a.__proto__ === Dog.prototype`은 `true`, 하지만 함수 자신의 `Dog.__proto__`는 `Function.prototype`이다. 둘은 다른 슬롯이다.

### 2. class도 동일한 체인

`class`는 위 패턴의 문법 설탕이다. 메서드는 자동으로 `prototype`에 올라간다.

```js
class Dog {
  constructor(name) { this.name = name; }
  bark() { return `${this.name}: 멍`; }   // Dog.prototype.bark 와 동일
}

const a = new Dog("초코");
Object.getPrototypeOf(a) === Dog.prototype; // true
Object.getOwnPropertyNames(Dog.prototype);  // ["constructor", "bark"]
```

`extends`는 두 단계 체인을 만든다.

```js
class Animal { eat() { return "냠"; } }
class Dog extends Animal { bark() { return "멍"; } }

const a = new Dog();
a.bark();  // "멍"  (Dog.prototype에서 발견)
a.eat();   // "냠"  (Dog.prototype에 없어 Animal.prototype까지 올라감)

// a ──▶ Dog.prototype ──▶ Animal.prototype ──▶ Object.prototype ──▶ null
```

### 3. Object.create로 직접 체인 연결

생성자 없이 부모를 지정해 객체를 만든다.

```js
const proto = { greet() { return "hi"; } };
const obj = Object.create(proto);   // obj.[[Prototype]] = proto

obj.greet();                        // "hi"
Object.getPrototypeOf(obj) === proto; // true

const bare = Object.create(null);   // 체인 없는 객체
bare.toString;                      // undefined (Object.prototype 미연결)
```

### 4. instanceof

`a instanceof Dog`는 "`Dog.prototype`이 `a`의 프로토타입 체인 어딘가에 있는가"를 검사한다.

```js
class Animal {}
class Dog extends Animal {}
const a = new Dog();

a instanceof Dog;     // true  (Dog.prototype 발견)
a instanceof Animal;  // true  (체인을 더 올라가 Animal.prototype 발견)
a instanceof Object;  // true  (Object.prototype까지 도달)

// 의사 동작:
function isInstance(obj, Ctor) {
  let p = Object.getPrototypeOf(obj);
  while (p !== null) {
    if (p === Ctor.prototype) return true;
    p = Object.getPrototypeOf(p);
  }
  return false;
}
```

## Pitfalls

체인 위의 프로퍼티에 대입하면 부모가 바뀌는 게 아니라 인스턴스에 가려쓰기(shadowing)가 생긴다.

```js
function Counter() {}
Counter.prototype.count = 0;

const c1 = new Counter();
const c2 = new Counter();
c1.count++;          // 읽기는 체인(0) → 쓰기는 c1 자신에 count:1 생성
c1.count;            // 1 (c1 자신)
c2.count;            // 0 (여전히 prototype)
```

참조 타입을 `prototype`에 두면 모든 인스턴스가 같은 객체를 공유해 의도치 않게 섞인다.

```js
function Box() {}
Box.prototype.items = [];   // 위험: 인스턴스 전체가 같은 배열 참조
const x = new Box(), y = new Box();
x.items.push(1);
y.items;                    // [1]  ← y도 영향받음
// 해결: this.items = [] 를 constructor 안에서 인스턴스마다 생성
```

`for...in`은 체인의 열거 가능한 프로퍼티까지 순회한다. 자신의 것만 보려면 `hasOwnProperty`나 `Object.keys`를 쓴다.

```js
const obj = Object.create({ inherited: 1 });
obj.own = 2;
for (const k in obj) console.log(k);     // "own", "inherited" 둘 다
Object.keys(obj);                         // ["own"] 만
```

`__proto__`는 표준이 아니라 호환용이다. 읽기/쓰기는 `Object.getPrototypeOf` / `Object.setPrototypeOf`를 쓴다. 또한 실행 중 `setPrototypeOf`로 체인을 바꾸면 JS 엔진의 인라인 캐시가 깨져 해당 객체 접근이 느려진다.

## 직접 확인

1. 콘솔에서 `const a = []`를 만들고 `a.__proto__ === Array.prototype`, `a.__proto__.__proto__ === Object.prototype`, `a.__proto__.__proto__.__proto__`를 차례로 평가해 `null`까지 몇 단계인지 직접 세어 본다. `a.map`이 어느 단계 객체에 있는지 `Object.getOwnPropertyNames`로 확인한다.
2. `function F(){}` 한 줄로 인스턴스 1만 개를 만드는 두 버전을 작성한다. (A) 메서드를 `this.fn = function(){}`로 constructor 안에 두기, (B) `F.prototype.fn = function(){}`로 두기. `performance.now()`로 생성 시간을 재고, `inst1.fn === inst2.fn` 결과를 비교해 공유 여부와 비용 차이를 측정한다.

## 스스로 답하기

1. `Dog.prototype`과 인스턴스의 `__proto__`(=`[[Prototype]]`)는 각각 무엇을 가리키며, 함수 자신의 `Dog.__proto__`는 왜 그와 다른가?
2. `obj.x = 1` 같은 쓰기 연산이 프로토타입 체인을 타지 않는 이유와, 그로 인해 발생하는 shadowing을 예로 설명할 수 있는가?
3. `a instanceof B`가 `false`를 반환하는 경우를 두 가지 이상 들 수 있는가?

## 내 프로젝트 연결

D3에서 `d3.select(".chart")`가 돌려주는 객체는 Selection 인스턴스다. `.attr`, `.data`, `.append` 같은 메서드는 그 객체 자신이 아니라 `Selection.prototype`에 정의돼 있고, 호출할 때마다 체인을 타고 찾아 쓴다. 콘솔에서 `const s = d3.select("body")`를 만든 뒤 `Object.getPrototypeOf(s)`와 `Object.getOwnPropertyNames(Object.getPrototypeOf(s))`를 찍으면 어떤 메서드가 prototype에 모여 있는지 직접 보인다. 같은 방식으로 axios 인스턴스, 직접 만든 유틸 class, React class 컴포넌트(`setState`는 `React.Component.prototype`에서 옴)의 메서드 출처도 추적해 본다. 본인이 매일 쓰는 객체의 메서드가 인스턴스에 있는지 prototype에 있는지 한 번 확인해 두면, 메서드를 못 찾는 `undefined is not a function` 류 에러를 체인 관점에서 빠르게 진단할 수 있다.
