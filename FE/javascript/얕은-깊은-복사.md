# 얕은 복사와 깊은 복사

> TL;DR: 얕은 복사는 한 단계만 새 객체로 만들고 중첩 객체는 참조를 공유하며, 깊은 복사는 중첩까지 전부 새 값으로 복제한다.

## What

원시 타입(number, string, boolean 등)을 변수에 대입하면 값 자체가 복사된다. 객체와 배열은 참조(메모리 주소)가 복사되므로, 대입한 변수 두 개가 같은 메모리를 가리킨다.

```js
const a = { x: 1 };
const b = a; // 참조 복사
b.x = 99;
console.log(a.x); // 99 (a와 b가 같은 객체)
```

얕은 복사(shallow copy)는 최상위 프로퍼티만 새 객체로 옮긴다. 값이 객체면 그 객체의 참조를 그대로 넣는다. 깊은 복사(deep copy)는 중첩된 객체까지 재귀적으로 새로 만들어, 원본과 어떤 참조도 공유하지 않는다.

## Why it matters

React, zustand, redux는 상태를 불변(immutable)으로 다룬다. 상태를 직접 수정하지 않고 새 객체를 만들어 교체해야 리렌더가 트리거된다. 이때 전개 연산자로 얕은 복사를 자주 쓴다. 중첩 상태를 얕게만 복사하면 안쪽 객체는 이전 상태와 같은 참조라서, 그 안을 수정하면 원본 상태까지 변하고 리렌더 판단(`Object.is` 비교)이 어긋난다. 면접에서 "전개 연산자가 깊은 복사냐"는 질문이 자주 나온다.

## How

전개 연산자와 `Object.assign`은 얕은 복사다.

```js
const orig = { name: "kim", info: { age: 20 } };

const copy1 = { ...orig };
const copy2 = Object.assign({}, orig);

copy1.name = "lee";        // 최상위는 분리됨
console.log(orig.name);    // "kim"

copy1.info.age = 99;       // 중첩 객체는 참조 공유
console.log(orig.info.age); // 99 (원본도 바뀜)
console.log(copy1.info === orig.info); // true
```

`JSON.parse(JSON.stringify(obj))`는 중첩까지 복제하는 깊은 복사 수단이다.

```js
const deep = JSON.parse(JSON.stringify(orig));
deep.info.age = 99;
console.log(orig.info.age); // 20 (원본 안 바뀜)
console.log(deep.info === orig.info); // false
```

`structuredClone`(2022년 이후 브라우저, Node 17+ 기본 제공)은 표준 깊은 복사 API다. JSON 방식이 버리는 `Date`, `Map`, `Set`, 순환 참조를 보존한다.

```js
const src = { d: new Date(), m: new Map([["k", 1]]) };
const clone = structuredClone(src);
console.log(clone.d instanceof Date); // true
console.log(clone.m.get("k"));        // 1
```

## Pitfalls

`JSON.parse(JSON.stringify())`는 직렬화 불가능한 값을 잃거나 변형한다.

```js
const data = {
  fn: () => 1,        // 함수: 사라짐
  u: undefined,       // undefined: 사라짐
  d: new Date(),      // Date: 문자열로 바뀜
  n: NaN,             // NaN, Infinity: null로 바뀜
};
console.log(JSON.parse(JSON.stringify(data)));
// { d: "2026-06-23T..." , n: null }  ← fn, u 키 없음, d는 string
```

순환 참조가 있으면 `JSON.stringify`가 예외를 던진다.

```js
const obj = {};
obj.self = obj;
JSON.parse(JSON.stringify(obj)); // TypeError: Converting circular structure to JSON
structuredClone(obj);            // 정상 동작 (순환 참조 보존)
```

중첩 상태를 얕게 복사하고 안쪽을 수정하면 원본이 변한다.

```js
const state = { user: { name: "kim" }, count: 0 };
const next = { ...state };
next.user.name = "lee"; // state.user.name도 "lee"로 변함
```

중첩 부분도 따로 복사해야 한다.

```js
const next = { ...state, user: { ...state.user, name: "lee" } };
```

## 직접 확인

1. 객체 하나를 만들고 전개 연산자로 복사한 뒤, `copy.중첩 === orig.중첩`을 `console.log`로 찍어 `true`가 나오는지 확인하고, 중첩 값을 바꿔 원본이 같이 변하는지 본다. 이어서 같은 객체에 함수와 `undefined`, `new Date()`를 넣고 `JSON.parse(JSON.stringify())`와 `structuredClone()` 결과를 나란히 출력해 어떤 키가 사라지거나 형이 바뀌는지 비교한다.
2. 1만 개 키를 가진 큰 객체를 만들고 `performance.now()`로 전개 연산자, `JSON.parse(JSON.stringify())`, `structuredClone()`의 소요 시간을 각각 측정해 비교한다.

## 스스로 답하기

- 전개 연산자로 객체를 복사했는데 중첩 객체를 수정하니 원본이 바뀌었다. 왜 그런가?
- `JSON.parse(JSON.stringify())`로 깊은 복사할 때 잃거나 변형되는 값에는 어떤 것들이 있는가?
- React/zustand에서 상태를 불변으로 업데이트할 때 깊은 복사 대신 필요한 경로만 얕게 복사하는 이유는 무엇인가?

## 내 프로젝트 연결

zustand/redux 상태를 업데이트할 때 전개 연산자로 얕은 복사를 쓰는 이유는, 바뀐 부분만 새 참조로 교체하고 나머지는 같은 참조로 유지해 리렌더 비교 비용을 줄이기 위해서다. 중첩 상태를 얕게만 복사하면 안쪽 객체가 이전 참조와 같아 `Object.is` 비교가 변경을 감지하지 못해 리렌더가 안 되거나, 반대로 안쪽을 직접 수정해 원본 상태가 변하는 버그가 생긴다. chatGraph 상태를 업데이트하는 코드에서 노드나 엣지 같은 중첩 구조를 어느 깊이까지 복사하는지 확인하고, 변경하는 경로마다 그 위 단계까지 새 객체로 만들고 있는지 점검해보라.
