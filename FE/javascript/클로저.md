# 클로저

> TL;DR: 함수가 선언된 위치의 렉시컬 환경을 참조로 들고 다녀서, 바깥 함수가 끝난 뒤에도 그 변수에 접근하는 함수와 환경의 조합.

## What

함수는 정의될 때 자신이 만들어진 위치의 스코프(렉시컬 환경)에 대한 참조를 갖는다. 이 참조는 함수가 다른 곳으로 반환되거나 전달돼도 유지된다. 그래서 바깥 함수의 실행이 끝나 호출 스택에서 사라진 뒤에도, 안쪽 함수는 바깥 변수를 읽고 쓸 수 있다.

스코프는 함수를 **어디서 선언했는지**로 정해진다(렉시컬 스코프). 호출 위치는 스코프 결정에 영향을 주지 않는다. 안쪽 함수가 살아 있는 한 그 환경의 변수는 가비지 컬렉션 대상에서 제외된다.

## Why it matters

- 모듈 패턴, 커스텀 훅, 이벤트 핸들러에서 외부 상태를 기억하는 기본 동작이 전부 클로저다.
- React 함수형 컴포넌트의 렌더마다 만들어지는 핸들러는 그 렌더 시점의 props/state를 클로저로 캡처한다. 오래된 값을 잡는 "stale closure" 버그가 여기서 나온다.
- `setTimeout`, 디바운스, 쓰로틀, 이벤트 리스너처럼 콜백이 나중에 실행되는 구조는 콜백이 정의 시점의 변수를 기억해야 동작한다.
- 면접에서 반복문+클로저 함정, 비공개 변수 구현, 메모리 누수 원인을 자주 묻는다.

## How

**1. 비공개 변수와 카운터**: 바깥 함수의 지역 변수 `count`는 외부에서 직접 접근할 수 없고, 반환된 함수를 통해서만 변경된다.

```js
function createCounter() {
  let count = 0; // 외부에서 직접 못 건드림
  return {
    increment() { count += 1; return count; },
    get() { return count; },
  };
}

const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.get();       // 2
// counter.count 는 undefined (캡슐화)
```

실행 순서: `createCounter()` 호출 → `count` 환경 생성 → 객체 반환 후 `createCounter` 실행 종료 → 그래도 `increment`가 `count` 환경을 참조하므로 살아남음 → 호출할 때마다 같은 `count`를 증가.

**2. 같은 팩토리, 독립된 환경**: 호출할 때마다 새 렉시컬 환경이 만들어져 상태가 섞이지 않는다.

```js
const a = createCounter();
const b = createCounter();
a.increment(); // 1
a.increment(); // 2
b.increment(); // 1  (a와 별개)
```

**3. 디바운스 구현**: 반환된 함수가 `timer`와 `delay`를 클로저로 기억한다. 호출이 들어올 때마다 이전 타이머를 지우고 다시 건다.

```js
function debounce(fn, delay) {
  let timer = null; // 클로저로 유지되는 상태
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

const onResize = debounce(() => console.log("resize 처리"), 200);
window.addEventListener("resize", onResize);
// 200ms 동안 추가 호출이 없을 때 한 번만 실행
```

**4. 쓰로틀 구현**: `last` 시각을 클로저로 기억해 일정 간격마다 한 번만 실행한다.

```js
function throttle(fn, interval) {
  let last = 0;
  return function (...args) {
    const now = Date.now();
    if (now - last >= interval) {
      last = now;
      fn.apply(this, args);
    }
  };
}
```

## Pitfalls

**반복문 + var (고전 함정)**: `var`는 함수 스코프라 콜백 세 개가 같은 `i` 하나를 공유한다. 루프가 끝난 뒤 콜백이 실행되므로 `i`는 이미 3이다.

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// 출력: 3, 3, 3
```

`let`은 블록 스코프라 반복마다 새 바인딩을 만든다. 각 콜백이 자기 반복의 `i`를 캡처한다.

```js
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// 출력: 0, 1, 2
```

`var`를 써야 하면 IIFE로 매 반복 새 스코프를 만든다.

```js
for (var i = 0; i < 3; i++) {
  (function (j) {
    setTimeout(() => console.log(j), 0);
  })(i);
}
// 출력: 0, 1, 2
```

**메모리 누수**: 큰 객체를 캡처한 클로저를 전역이나 오래 사는 자료구조에 보관하면, 그 객체가 GC되지 않는다. 더 안 쓰는 리스너는 `removeEventListener`로 끊는다.

**stale closure (React)**: 의존성 배열에서 값을 빼면 핸들러가 오래된 state를 기억한다.

```js
useEffect(() => {
  const id = setInterval(() => {
    console.log(count); // count를 빼면 항상 초기값만 찍힘
  }, 1000);
  return () => clearInterval(id);
}, [count]); // count를 의존성에 넣어야 최신값 캡처
```

## 직접 확인

1. `createCounter`를 두 번 호출해 인스턴스 두 개를 만들고, 한쪽만 5번 increment한 뒤 다른 쪽 `get()`이 0인지 콘솔로 확인하라. 환경이 분리되는 것을 눈으로 본다.
2. 위 `var`/`let` 반복문 예제를 그대로 브라우저 콘솔에 붙여 실행하고 출력을 비교하라. 그다음 `let` 버전을 디버거로 멈춰 각 반복의 `i`가 별도 바인딩인지 Scope 패널에서 확인하라.

## 스스로 답하기

1. 바깥 함수가 이미 반환됐는데 안쪽 함수가 바깥 변수를 읽을 수 있는 이유를 렉시컬 환경 관점에서 설명해 보라.
2. `var`로 만든 반복문 콜백이 모두 같은 값을 출력하는 이유와, `let`이 이를 고치는 메커니즘의 차이는?
3. 디바운스 함수에서 `timer` 변수가 매 호출마다 초기화되지 않고 유지되는 이유는?

## 내 프로젝트 연결

chatGraph의 optimistic 라우팅에서, 응답이 오기 전 만든 temp ID를 핸들러가 클로저로 캡처해 나중에 실제 ID로 치환했다. 무한 스크롤의 디바운스 콜백도 `timer`와 콜백을 클로저로 기억해 마지막 스크롤만 처리했다. 두 코드에서 어떤 변수가 클로저로 캡처됐는지, 그 변수가 어느 시점에 생성되고 언제까지 살아 있었는지 직접 짚어 보라.
