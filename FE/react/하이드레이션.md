# 하이드레이션

> TL;DR: 서버가 미리 만든 정적 HTML에 클라이언트 React가 같은 컴포넌트 트리를 다시 렌더해 매칭하고, 기존 DOM 노드를 재사용하면서 이벤트 핸들러와 상태를 연결하는 과정.

## What

SSR은 서버에서 컴포넌트를 HTML 문자열로 렌더해 브라우저로 보낸다. 이 HTML은 화면에 보이지만 `onClick` 같은 이벤트가 붙어 있지 않아 동작하지 않는다. 하이드레이션은 클라이언트에서 `hydrateRoot`가 같은 트리를 한 번 더 렌더하고, 서버가 만든 DOM 노드를 새로 생성하지 않고 그대로 재사용하면서 이벤트 핸들러와 `useState`·`useEffect` 같은 상태를 연결하는 단계다.

동작 순서:

1. 서버가 컴포넌트를 렌더해 HTML을 응답한다.
2. 브라우저가 HTML을 그려 화면을 보여준다 (이 시점에는 클릭이 안 먹는다).
3. JS 번들이 로드되고 `hydrateRoot(container, <App />)`가 실행된다.
4. React가 클라이언트에서 트리를 렌더하며 기존 DOM과 1:1로 매칭한다.
5. 매칭에 성공하면 이벤트 핸들러를 붙이고 앱이 상호작용 가능해진다.

`createRoot`는 빈 컨테이너에 DOM을 새로 만들고, `hydrateRoot`는 이미 존재하는 서버 HTML에 붙는다는 점이 다르다.

## Why it matters

SSR/SSG를 쓰는 Next.js 프로젝트의 기본 렌더 모델이다. 서버 렌더 결과와 클라이언트 첫 렌더 결과가 다르면 hydration mismatch 에러가 콘솔에 찍히고, React가 해당 서브트리를 버리고 클라이언트에서 다시 렌더한다. 이때 깜빡임이 생기거나 화면이 잠깐 어긋난다.

면접에서는 "SSR HTML이 보이는데 왜 버튼이 안 눌리나", "hydration mismatch는 왜 나는가", "`use client`는 무엇을 가르는 경계인가"를 묻는다. App Router에서 서버 컴포넌트와 클라이언트 컴포넌트를 나누는 기준이 바로 하이드레이션 대상 여부다.

## How

기본 진입점. 서버 HTML이 들어 있는 컨테이너에 `hydrateRoot`로 붙인다.

```tsx
import { hydrateRoot } from "react-dom/client";
import App from "./App";

// 서버가 #root 안에 이미 HTML을 그려 둔 상태
hydrateRoot(document.getElementById("root")!, <App />);
```

Next.js App Router에서 `use client`로 클라이언트 경계를 선언한다. 이 파일과 여기서 import하는 하위 컴포넌트가 하이드레이션 대상이 된다.

```tsx
"use client";

import { useState } from "react";

export default function Counter() {
  // useState, onClick은 클라이언트에서만 동작 → 하이드레이션 후 연결됨
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

서버 컴포넌트(`use client` 없음)는 서버에서만 렌더되어 HTML로 직렬화되고, 하이드레이션 대상이 아니다. 그래서 `onClick`이나 `useState`를 쓰면 빌드 에러가 난다.

```tsx
// 서버 컴포넌트 (기본값). 데이터만 가져와 HTML 생성.
export default async function Page() {
  const data = await fetch("https://api.example.com/posts").then((r) => r.json());
  return (
    <main>
      <h1>{data.title}</h1>
      <Counter /> {/* 상호작용이 필요한 부분만 클라이언트 컴포넌트로 분리 */}
    </main>
  );
}
```

실행 결과: `<h1>`은 서버에서 완성돼 즉시 보이고, `<Counter />`만 JS 번들 로드 후 하이드레이션되어 클릭이 동작한다.

## Pitfalls

서버와 클라이언트의 첫 렌더 결과가 달라지는 코드가 mismatch의 주원인이다.

`window`·`localStorage`처럼 서버에 없는 객체에 접근하면, 서버는 한 쪽 분기를, 클라이언트는 다른 분기를 렌더해 트리가 어긋난다.

```tsx
"use client";

// 나쁜 예: 서버에는 window가 없어 false, 클라이언트에서는 true → mismatch
function Bad() {
  const isWide = typeof window !== "undefined" && window.innerWidth > 768;
  return <div>{isWide ? "데스크톱" : "모바일"}</div>;
}

// 좋은 예: 첫 렌더는 서버와 동일하게, 브라우저 값은 useEffect 이후에 반영
import { useState, useEffect } from "react";
function Good() {
  const [isWide, setIsWide] = useState(false); // 서버·클라 첫 렌더 동일
  useEffect(() => {
    setIsWide(window.innerWidth > 768); // 하이드레이션 후 갱신
  }, []);
  return <div>{isWide ? "데스크톱" : "모바일"}</div>;
}
```

`Date.now()`·`Math.random()`·`new Date().toLocaleString()`처럼 렌더 시점마다 값이 바뀌는 코드도 서버 렌더 시각과 클라이언트 렌더 시각이 달라 mismatch를 만든다. 이런 값은 `useEffect` 안에서 set하거나, 서버에서 계산한 값을 props로 내려보내 고정한다.

`<p>` 안에 `<div>`를 넣는 식의 잘못된 HTML 중첩도 mismatch를 부른다. 브라우저가 이런 구조를 자동으로 교정하면서 서버가 보낸 HTML 구조를 바꿔 React 트리와 매칭이 깨지기 때문이다.

## 직접 확인

- Next.js App Router 프로젝트를 만들고, 클라이언트 컴포넌트 안에서 `new Date().toLocaleTimeString()`을 렌더해 본다. 브라우저 콘솔에 hydration mismatch 경고가 뜨는지 확인하고, 그 값을 `useState`+`useEffect`로 옮긴 뒤 경고가 사라지는지 비교한다.
- 같은 컴포넌트를 띄운 상태에서 Network 탭에서 JS 번들 요청을 throttling(Slow 3G)으로 늦춰 본다. HTML은 먼저 보이지만 버튼 클릭이 몇 초간 무반응이다가, 번들 로드 완료 시점부터 동작하기 시작하는 구간을 직접 관찰한다.

## 스스로 답하기

- SSR로 받은 HTML이 화면에 다 보이는데도 버튼 클릭이 한동안 동작하지 않는 이유는 무엇인가?
- hydration mismatch가 발생하면 React는 그 서브트리를 어떻게 처리하며, 사용자 화면에는 어떤 현상으로 나타나는가?
- `use client`를 선언한 컴포넌트와 선언하지 않은 서버 컴포넌트는 빌드와 런타임에서 각각 어떻게 다르게 처리되는가?

## 내 프로젝트 연결

MindGraph에서 D3로 그래프를 그릴 때 SSR과 충돌했다. D3는 `d3.select()`로 DOM 노드를 직접 잡아 `append`·`attr`로 SVG를 조작하는데, 이 조작은 React의 가상 DOM 바깥에서 실제 DOM을 바꾼다. 서버 렌더 시점에는 `document`가 없어 D3 코드가 실행될 수 없고, 클라이언트에서 D3가 DOM을 바꾼 뒤 React가 하이드레이션하면 서버 HTML과 D3가 만든 실제 DOM이 달라 mismatch가 났다.

해결은 그래프 엔진을 `use client` 전용 레이어로 격리한 것이다. D3를 호출하는 컴포넌트 최상단에 `"use client"`를 선언해 서버 렌더 대상에서 빼고, 컨테이너 `ref`만 서버 HTML로 내보낸 뒤 실제 D3 조작은 `useEffect` 안에서만 실행했다. 서버는 빈 컨테이너만 그리고, D3는 하이드레이션이 끝난 클라이언트에서만 DOM을 채우므로 서버/클라이언트 렌더 결과가 어긋나지 않는다.
