# 리렌더링은 언제 왜 일어나는가

> TL;DR: 컴포넌트는 state 변경, props 변경, 부모 리렌더 중 하나가 생기면 다시 렌더된다. 부모가 리렌더되면 자식은 props가 같아도 기본적으로 함께 리렌더된다.

## What

렌더는 React가 컴포넌트 함수를 호출해 화면에 그릴 결과(JSX)를 계산하는 단계다. 리렌더는 같은 컴포넌트 함수를 다시 호출하는 것을 말한다. 트리거는 3가지다.

1. **state 변경**: `useState`/`useReducer`의 setter가 호출되고 새 값이 이전 값과 다르면(`Object.is` 비교) 해당 컴포넌트를 리렌더 큐에 넣는다.
2. **props 변경**: 부모가 새 props를 내려주면 자식이 리렌더된다. 정확히는 "부모가 리렌더되면서 새 props를 만든 결과"다.
3. **부모 리렌더**: 부모 컴포넌트가 리렌더되면 그 아래 자식 트리는 props가 동일해도 전부 다시 렌더 함수가 호출된다.

리렌더는 렌더 단계(함수 호출 + 가상 트리 비교)이고, 실제 DOM 변경은 그 뒤 커밋 단계에서만 일어난다. 함수가 다시 호출됐다고 DOM이 항상 바뀌는 것은 아니다.

## Why it matters

리스트가 길거나 트리가 깊은 화면에서 상위 한 곳의 state 변경이 하위 수백 개 컴포넌트의 렌더 함수를 전부 호출시켜 입력 지연이나 프레임 드랍이 생긴다. 면접에서는 "부모가 리렌더되면 자식도 리렌더되는가", "`React.memo`로 막을 수 있는가", "왜 memo를 걸었는데도 자식이 계속 리렌더되는가"가 자주 나온다. 마지막 질문의 답은 대개 객체/함수 props가 매 렌더 새로 생기는 함정이다.

## How

**state 변경으로 리렌더**

```tsx
function Counter() {
  const [count, setCount] = useState(0);
  console.log("Counter 렌더");
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

버튼 클릭 → `setCount` 호출 → 새 값(이전과 다름) → Counter 리렌더 → 콘솔에 "Counter 렌더" 출력. 같은 값으로 `setCount(count)`를 호출하면 React가 `Object.is`로 같다고 판단해 리렌더를 건너뛴다. 다만 React가 해당 컴포넌트를 한 번 더 렌더한 뒤 결과가 같음을 확인하고 자식 렌더와 DOM 변경을 생략하는 경우도 있으므로, 항상 함수 호출 자체가 0번이라고 단정하지는 않는다.

**부모 리렌더 → 자식도 리렌더**

```tsx
function Parent() {
  const [n, setN] = useState(0);
  return (
    <>
      <button onClick={() => setN(n + 1)}>{n}</button>
      <Child label="고정 텍스트" />
    </>
  );
}

function Child({ label }: { label: string }) {
  console.log("Child 렌더");
  return <p>{label}</p>;
}
```

실행 순서: 버튼 클릭 → Parent의 state 변경 → Parent 리렌더 → Parent 함수가 다시 호출되며 `<Child .../>`도 다시 평가됨 → `label`이 같아도 Child 렌더 함수 호출 → "Child 렌더" 출력. props가 안 바뀌어도 자식 함수는 호출된다.

**React.memo로 부모 리렌더 전파 끊기**

```tsx
const Child = React.memo(function Child({ label }: { label: string }) {
  console.log("Child 렌더");
  return <p>{label}</p>;
});
```

`React.memo`는 이전 props와 새 props를 얕게 비교(`Object.is`)해서 모두 같으면 자식 렌더 함수 호출을 건너뛴다. 위 예제처럼 `label="고정 텍스트"` 문자열이면 매번 같으므로, Parent가 100번 리렌더돼도 Child는 1번만 렌더된다.

**useMemo / useCallback로 참조 안정화**

```tsx
function Parent() {
  const [n, setN] = useState(0);

  // useCallback 없으면 매 렌더 새 함수 → memo(Child) 깨짐
  const handleClick = useCallback(() => console.log("click"), []);

  // useMemo 없으면 매 렌더 새 객체 → memo(Child) 깨짐
  const style = useMemo(() => ({ color: "red" }), []);

  return (
    <>
      <button onClick={() => setN(n + 1)}>{n}</button>
      <Child onClick={handleClick} style={style} />
    </>
  );
}
```

`useCallback`은 함수 참조를, `useMemo`는 계산 결과 참조를 의존성 배열이 안 바뀌는 한 동일하게 유지한다. 그래야 `React.memo(Child)`의 얕은 비교가 통과해 자식 리렌더가 끊긴다.

**한계**: memo/useMemo/useCallback은 비교 비용과 메모리를 추가로 쓴다. 자식이 가볍거나 props가 거의 매번 바뀌면 비교만 하고 결국 리렌더되므로 오히려 손해다. 모든 자식에 기계적으로 memo를 거는 것은 권장되지 않는다.

## Pitfalls

**객체/함수 props가 매 렌더 새로 생겨 memo가 깨짐**

```tsx
const Child = React.memo(function Child({ style, onClick }) {
  console.log("Child 렌더");
  return <button style={style} onClick={onClick}>버튼</button>;
});

function Parent() {
  const [n, setN] = useState(0);
  return (
    <>
      <button onClick={() => setN(n + 1)}>{n}</button>
      {/* 매 렌더마다 새 객체·새 함수 리터럴 생성 */}
      <Child style={{ color: "red" }} onClick={() => console.log("x")} />
    </>
  );
}
```

`{ color: "red" }`와 `() => ...`는 렌더할 때마다 새 참조다. `React.memo`의 얕은 비교에서 `이전.style Object.is 새.style`이 `false`이므로 memo를 걸어도 Child가 매번 리렌더된다. 콘솔에 클릭마다 "Child 렌더"가 찍힌다. 해결은 위 How의 `useMemo`/`useCallback`으로 참조를 고정하는 것이다.

**부모가 안 바뀐 자식까지 리렌더하는 걸 "state 위치"로 막기**

state를 쓰는 부분만 작은 컴포넌트로 분리하면, memo 없이도 state 변경 범위가 그 컴포넌트로 좁혀져 형제 트리는 리렌더되지 않는다.

```tsx
function Parent() {
  return (
    <>
      <Counter />        {/* state는 여기 안에 격리 */}
      <HeavyTree />      {/* Counter가 바뀌어도 영향 없음 */}
    </>
  );
}
```

## 직접 확인

1. 위 "부모 리렌더 → 자식도 리렌더" 예제를 그대로 만들고, Child에 `console.log`를 넣은 뒤 버튼을 5번 눌러 콘솔에 "Child 렌더"가 몇 번 찍히는지 센다. 그다음 `React.memo`로 Child를 감싸고 다시 5번 눌러 횟수가 1번으로 줄어드는지 확인한다.
2. memo(Child)에 `style={{ color: "red" }}`를 props로 추가하고 버튼을 5번 눌러본다. memo가 깨져 "Child 렌더"가 다시 5번 찍히는지 보고, `useMemo`로 style을 감싼 뒤 1번으로 돌아오는지 측정한다.

## 스스로 답하기

1. 부모가 리렌더될 때 props가 전혀 바뀌지 않은 자식의 렌더 함수는 호출되는가, 호출되지 않는가? 그 이유는?
2. `React.memo`로 자식을 감쌌는데도 부모가 리렌더될 때마다 자식이 계속 리렌더된다면, 가장 흔한 원인 2가지는 무엇인가?
3. 렌더 단계와 커밋 단계는 어떻게 다른가? 렌더 함수가 다시 호출됐을 때 실제 DOM은 항상 바뀌는가?

## 내 프로젝트 연결

chatGraph에서 843줄짜리 단일 훅을 355줄로 쪼개 분해할 때, 한 훅 안에 모든 state가 모여 있으면 그중 하나만 바뀌어도 그 훅을 쓰는 컴포넌트 전체가 리렌더됐다. 책임별로 훅을 나누고 state를 실제로 쓰는 컴포넌트 가까이로 내리면서 리렌더 범위가 좁아졌다. React DevTools Profiler의 "Highlight updates when components render"를 켜고 한 번 입력해서 화면 어느 영역이 깜빡이는지 본인 컴포넌트로 직접 측정하라. 분해 전후로 리렌더되는 컴포넌트 수와 렌더 시간이 얼마나 달라지는지 기록한다.
