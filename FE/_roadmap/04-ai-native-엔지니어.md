# AI-native 엔지니어

> TL;DR: AI-native는 프롬프트를 더 잘 쓰는 능력이 아니라, 본인이 아키텍트 자리에 앉은 채 실행만 위임하는 작업 방식이다. AI가 코드를 더 많이 쓸수록 검증·명세·분해·판단의 가치가 올라간다.

생산성이 오른 것(병렬로 코드를 만드는 것)은 버리지 않는다. 거기에 아래 다섯 가지를 얹어 운전석을 되찾는다.

## 1. 검증력

AI가 코드를 많이 쓸수록 병목은 "이게 맞는가"로 옮겨간다. 읽지 못하는 코드는 본인 코드가 아니다. 머지 전 한 줄씩 설명([[01-습관과-원칙]] 원칙 1), 테스트 작성, 엣지 케이스 상상이 핵심 작업이 된다.

실천: 변경마다 AI에 "회의적인 리뷰어처럼 이 diff를 공격해봐"를 시키고, 지적받은 부분을 직접 실행해 확인한다.

## 2. 명세력

구현 전에 입력·출력·엣지 케이스·실패 시 동작을 글로 적고 시작한다. 모호한 요청은 모호한 결과를 낳는다. "비전공 바이브 코더 수준 요청"이라고 본인이 느낀 지점이 여기다. 방법은 [[../../AI-Native/spec-driven/sdd-프레임워크]].

## 3. 분해력

큰 작업을 검증 가능한 작은 조각으로 쪼개 하나씩 시키고 확인한다. 한 번에 큰 덩어리를 시키고 통째로 믿는 패턴을 끊는다. 각 조각은 "끝났는지 어떻게 확인할지"가 분명해야 한다.

## 4. AI를 튜터로 쓴다

"짜줘" 대신 "왜 이렇게 했어 / 트레이드오프는 뭐야 / 이걸 깨뜨리는 입력은 / 다른 방법과 비교해줘"를 묻는다. 답을 받으면 직접 실행해 검증한다. 같은 시간으로 코드와 이해를 함께 얻는다.

## 5. 판단력과 취향

무엇을 만들지, 어떤 아키텍처와 추상화가 맞는지 정하는 일은 AI가 아직 대신 못 한다. 트레이드오프 선택을 위임하지 않고 직접 내리는 연습을 [[03-프로젝트-소유권]]에서 한다.

## AI 의존 신호 (보이면 원칙으로 복귀)

- 짠 코드를 설명하지 못한 채 머지한다
- 에러를 가설 없이 AI에 먼저 던진다
- 라이브러리나 패턴을 고른 이유를 모른다

이 신호가 보이면 [[01-습관과-원칙]]으로 돌아가 그 작업을 다시 한다.
