# 브라우저 렌더링 파이프라인

> TL;DR: 브라우저는 DOM·CSSOM → Render Tree → Layout → Paint → Composite 순으로 화면을 그리며, transform/opacity만 바꾸면 앞 단계를 건너뛰고 Composite만 다시 한다.

## What

브라우저가 HTML/CSS/JS를 받아 픽셀로 그리기까지 거치는 단계의 순서를 critical rendering path라고 한다.

1. **HTML 파싱 → DOM**: 바이트를 토큰으로, 토큰을 노드로 변환해 DOM 트리를 만든다.
2. **CSS 파싱 → CSSOM**: 스타일시트를 파싱해 CSSOM 트리를 만든다. 셀렉터의 상속·우선순위가 여기서 계산된다.
3. **Render Tree**: DOM과 CSSOM을 합쳐 화면에 그릴 노드만 모은다. `display: none`은 제외되고, `visibility: hidden`은 포함된다.
4. **Layout (reflow)**: 각 노드의 위치와 크기를 픽셀 단위로 계산한다. 뷰포트 크기, 폰트, box 모델에 따라 좌표가 정해진다.
5. **Paint**: 계산된 박스를 색·테두리·그림자·텍스트 등 픽셀로 채운다. 여러 레이어로 나뉠 수 있다.
6. **Composite**: 페인트된 레이어들을 GPU에서 합성해 최종 화면을 만든다.

`width`, `height`, `margin`, `padding`이나 positioned 요소의 `top`/`left` 같은 기하 속성을 바꾸면 Layout부터 다시 한다(reflow). `color`, `background` 등 색만 바꾸면 Layout을 건너뛰고 Paint부터 한다(repaint). `transform`, `opacity`는 별도 레이어로 분리되면 Composite만 다시 한다.

## Why it matters

스크롤·애니메이션이 60fps를 유지하려면 프레임당 16.7ms 안에 한 사이클을 끝내야 한다. reflow는 영향받는 노드뿐 아니라 하위·후속 노드의 좌표를 재계산하므로 비용이 가장 크다. 면접에서는 "transform과 top 중 어느 쪽으로 애니메이션해야 하느냐", "강제 동기 레이아웃이 무엇이냐", "레이아웃 스래싱을 어떻게 막느냐"를 자주 묻는다.

## How

**reflow를 유발하는 속성 vs Composite만 유발하는 속성**

```js
// reflow 발생: positioned 요소의 left를 바꾸면 매 프레임 Layout 재계산
el.style.position = "absolute";
el.style.left = x + "px";

// Composite만 발생: transform은 레이어 합성만 다시 함
el.style.transform = `translateX(${x}px)`;
```

`transform`/`opacity`는 메인 스레드의 Layout/Paint를 건너뛰고 컴포지터 스레드에서 처리되므로 JS가 바빠도 부드럽게 움직인다.

**읽기와 쓰기를 분리해 reflow 횟수 줄이기**

```js
// 나쁨: 쓰기 → 읽기 → 쓰기 → 읽기 순서가 매번 동기 레이아웃을 강제
for (const el of items) {
  el.style.width = el.offsetWidth + 10 + "px"; // 쓴 직후 읽으면 강제 reflow
}

// 좋음: 모든 읽기를 먼저, 모든 쓰기를 나중에
const widths = items.map((el) => el.offsetWidth); // 읽기 일괄
items.forEach((el, i) => {
  el.style.width = widths[i] + 10 + "px";        // 쓰기 일괄
});
```

**여러 DOM 변경을 한 번의 reflow로 묶기**

```js
const frag = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
  const li = document.createElement("li");
  li.textContent = `node ${i}`;
  frag.appendChild(li); // 아직 문서에 없으므로 reflow 없음
}
list.appendChild(frag); // 한 번만 reflow
```

## Pitfalls

**강제 동기 레이아웃**: 스타일을 쓴 직후 `offsetHeight`, `offsetTop`, `getBoundingClientRect()`, `scrollTop`, `getComputedStyle()` 같은 기하 값을 읽으면, 브라우저는 정확한 값을 주려고 그 자리에서 Layout을 즉시 실행한다.

```js
el.classList.add("expanded");        // 쓰기 → 레이아웃 dirty 표시
const h = el.offsetHeight;            // 읽기 → 여기서 강제 동기 reflow
el.classList.add("done");            // 쓰기 → 다시 dirty
const top = el.getBoundingClientRect().top; // 읽기 → 또 강제 reflow
```

위 코드는 한 함수 안에서 reflow를 두 번 강제한다. 이게 루프 안에 있으면 레이아웃 스래싱이 된다.

```js
// 레이아웃 스래싱: N개 요소에 대해 N번 강제 reflow
boxes.forEach((box) => {
  box.style.height = container.offsetHeight + "px"; // 읽기+쓰기 교차
});
```

**`will-change` 과용**: `will-change: transform`을 모든 요소에 걸면 레이어가 과도하게 생겨 메모리를 잡아먹고 오히려 느려진다. 실제 애니메이션 직전에만 켜고 끝나면 제거한다.

## 직접 확인

1. `for` 루프 안에서 `el.style.width`를 쓴 직후 `el.offsetWidth`를 읽는 코드를 100개 노드에 돌려보고, DevTools Performance 탭으로 녹화해 보라. "Recalculate Style"과 "Layout" 막대가 보라색으로 반복되는지(레이아웃 스래싱) 확인하고, 읽기/쓰기를 분리한 버전과 총 Layout 시간을 비교하라.
2. 같은 좌표 이동을 `left`로 한 버전과 `transform: translateX`로 한 버전을 각각 만들어 Performance로 녹화하라. 어느 쪽이 메인 스레드에서 Layout/Paint를 일으키고, 어느 쪽이 Composite만 도는지 프레임 차트로 비교하라.

## 스스로 답하기

1. reflow와 repaint의 차이는 무엇이고, 각각 어떤 CSS 속성 변경이 유발하는가?
2. `transform: translateX(100px)`가 `left: 100px`보다 애니메이션에서 부드러운 이유를 파이프라인 단계로 설명할 수 있는가?
3. 강제 동기 레이아웃은 코드의 어떤 패턴에서 발생하며, 어떻게 없앨 수 있는가?

## 내 프로젝트 연결

MindGraph에서 D3가 DOM(SVG)을 직접 조작할 때, 노드 좌표를 매 tick마다 `attr("x")`로 쓰고 곧바로 위치를 읽으면 강제 동기 레이아웃이 누적된다. xAB 무한 스크롤로 노드가 늘어날수록 한 번의 Layout이 훑어야 할 노드 수가 비례해 커진다. Performance 탭으로 본인 페이지를 녹화해, 스크롤 중 Layout 막대가 프레임당 16.7ms를 넘는 구간이 있는지 측정하고 reflow를 유발하는 코드 줄을 특정해 보라.
