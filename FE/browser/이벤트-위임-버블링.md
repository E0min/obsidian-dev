# 이벤트 위임과 버블링

> TL;DR: 이벤트는 캡처 → 타깃 → 버블 3단계로 전파되며, 이 버블링을 이용해 부모에 핸들러 하나만 등록하고 `event.target`으로 실제 클릭된 자식을 식별하는 패턴이 이벤트 위임이다.

## What

DOM 이벤트는 발생 지점에서 끝나지 않고 트리를 따라 두 방향으로 전파된다. 단계는 `event.eventPhase` 값으로 구분된다.

1. **캡처 단계** (`eventPhase === 1`): document → 타깃의 부모까지 위에서 아래로 내려온다.
2. **타깃 단계** (`eventPhase === 2`): 이벤트가 실제 발생한 요소에 도달한다.
3. **버블 단계** (`eventPhase === 3`): 타깃 → document까지 아래에서 위로 올라간다.

`addEventListener(type, handler)`의 기본값은 버블 단계에서 실행한다. 세 번째 인자를 `true`(또는 `{ capture: true }`)로 주면 캡처 단계에서 실행한다.

- `event.target`: 이벤트가 처음 발생한 요소. 전파 중에도 바뀌지 않는다.
- `event.currentTarget`: 지금 핸들러가 붙어 있는 요소. 버블/캡처로 전파되며 단계마다 바뀐다. (`this`와 동일)

**이벤트 위임**: 자식마다 핸들러를 다는 대신 공통 부모에 핸들러 하나를 등록하고, 버블링으로 올라온 이벤트의 `event.target`을 검사해 어느 자식에서 발생했는지 판별하는 기법.

## Why it matters

- 리스트 항목이 수백~수천 개면 항목마다 리스너를 다는 것은 메모리와 등록 비용을 키운다. 위임은 리스너 1개로 처리한다.
- 무한 스크롤·필터·정렬로 DOM이 동적으로 추가·삭제될 때, 항목마다 다는 방식은 추가 시점마다 다시 `addEventListener`를 호출해야 한다. 부모에 위임하면 나중에 들어온 자식도 자동으로 처리된다.
- 면접에서 `target` vs `currentTarget` 차이, `stopPropagation`이 막는 것과 못 막는 것, 캡처/버블 순서를 자주 묻는다.

## How

**3단계 전파 순서 확인**:
```js
// HTML: <div id="outer"><button id="btn">클릭</button></div>
const outer = document.getElementById("outer");
const btn = document.getElementById("btn");

outer.addEventListener("click", () => console.log("outer 캡처"), true);  // 캡처
outer.addEventListener("click", () => console.log("outer 버블"));         // 버블
btn.addEventListener("click", () => console.log("btn 타깃"));

// btn 클릭 시 콘솔 출력 순서:
// outer 캡처  → outer 버블  → btn 타깃 (X)
// 실제: "outer 캡처" → "btn 타깃" → "outer 버블"
```
캡처는 위에서 내려오므로 `outer`가 먼저, 타깃에 도달한 뒤, 버블로 다시 `outer`가 마지막에 찍힌다.

**target vs currentTarget**:
```js
outer.addEventListener("click", (e) => {
  console.log(e.target.id);        // "btn" (실제 클릭된 요소)
  console.log(e.currentTarget.id); // "outer" (핸들러가 붙은 요소)
});
```

**이벤트 위임**:
```js
const list = document.getElementById("list");
// <ul id="list"><li data-id="1">A</li><li data-id="2">B</li></ul>

list.addEventListener("click", (e) => {
  const li = e.target.closest("li");   // 클릭 지점에서 가장 가까운 li 탐색
  if (!li || !list.contains(li)) return; // 리스트 밖이면 무시
  console.log("선택된 id:", li.dataset.id);
});
```
`closest("li")`를 쓰면 `<li>` 내부의 `<span>`·아이콘을 클릭해도 부모 `<li>`를 정확히 찾는다.

**stopPropagation / preventDefault**:
```js
btn.addEventListener("click", (e) => {
  e.stopPropagation();  // 이 이벤트의 추가 전파(버블/캡처)를 중단
  e.preventDefault();   // 브라우저 기본 동작 취소 (예: <a> 이동, submit)
});
```
두 메서드는 역할이 다르다. `stopPropagation`은 전파를 끊고, `preventDefault`는 기본 동작만 막는다. 한쪽이 다른 쪽을 대체하지 않는다.

## Pitfalls

- **위임을 쓰면서 `stopPropagation`을 남발하면 위임이 깨진다.** 자식에서 전파를 끊으면 부모의 위임 핸들러까지 이벤트가 올라오지 못한다.
```js
li.addEventListener("click", (e) => e.stopPropagation()); // 부모 위임 핸들러가 안 탄다
```

- **`target`을 `currentTarget`처럼 쓰는 실수**: 위임 핸들러에서 `e.target`은 실제 클릭된 가장 깊은 요소다. `<li>` 안에 `<span>`이 있으면 `e.target`이 `<span>`이 되어 `dataset.id`가 undefined가 된다. `closest()`로 끌어올려야 한다.

- **`focus`·`blur`·`mouseenter`·`mouseleave`는 버블링하지 않는다.** 위임이 필요하면 버블링되는 `focusin`/`focusout`, `mouseover`/`mouseout`을 쓴다.
```js
form.addEventListener("focusin", (e) => { /* 동작 */ });  // focus는 안 됨
```

- **캡처 단계로 등록한 핸들러는 `stopPropagation`으로 타깃 핸들러까지 막을 수 있다.** 등록 단계(캡처 vs 버블)를 헷갈리면 의도치 않게 하위 이벤트가 사라진다.

## 직접 확인

1. `<ul>` 안에 `<li>` 5개를 만들고, (a) 항목마다 `addEventListener`를 다는 코드와 (b) `<ul>`에 위임 핸들러 1개를 다는 코드를 각각 작성한다. 그다음 JS로 `<li>` 1개를 동적으로 `appendChild`했을 때, 두 방식에서 새 항목의 클릭이 동작하는지 콘솔로 비교한다.
2. 같은 요소에 캡처 핸들러와 버블 핸들러를 모두 달고, 중첩 div 3겹을 클릭해 `e.eventPhase` 값과 콘솔 출력 순서를 직접 기록한다. 중간 div에서 `stopPropagation()`을 호출했을 때 출력이 어떻게 끊기는지도 확인한다.

## 스스로 답하기

1. `event.target`과 `event.currentTarget`은 위임 핸들러 안에서 각각 무엇을 가리키며, 왜 위임에서는 `target`을 그대로 쓰면 안 되는가?
2. `stopPropagation()`과 `preventDefault()`의 차이는 무엇이고, 위임 패턴에서 `stopPropagation()`이 일으키는 문제는 무엇인가?
3. `focus` 이벤트로 위임이 안 되는 이유는 무엇이고, 어떤 이벤트로 대체하는가?

## 내 프로젝트 연결

xAB 무한 스크롤 리스트는 스크롤마다 항목이 동적으로 추가된다. 항목마다 핸들러를 달면 추가 시점마다 등록 코드가 다시 실행되고 리스너 수가 누적된다. 부모 컨테이너에 위임 핸들러 1개를 두면 나중에 들어온 항목도 별도 등록 없이 클릭이 잡힌다. 본인 리스트 컴포넌트가 항목 단위로 `onClick`을 다는지, 컨테이너 한 곳에서 `e.target`/`closest`로 처리하는지 코드를 열어 확인하고, 동적 추가 항목의 클릭이 정상 동작하는지 측정해보라.
