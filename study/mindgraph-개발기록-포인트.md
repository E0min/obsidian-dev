# MindGraph 개발 기록 포인트

> TL;DR: MindGraph는 영상 4중점의 '서비스 운영 경험'(기획→설계→구현→배포→운영→개선)을 증명할 사이드 프로젝트다. 개발하며 의사결정·측정·트러블슈팅·운영 가설을 그때그때 남기면, 그 기록이 곧 이력서의 숫자 성과이자 면접 답변([[03-프로젝트-소유권]])이 된다.

지나간 뒤 복기하면 디테일이 날아간다. 아래를 개발 중에 바로 남긴다.

## 1. 라이프사이클 전 과정을 일지로

큰돌이 말한 "처음부터 끝까지 만든 경험"의 증거. 각 단계에서 내린 결정과 근거를 짧게 남긴다.

- 기획: 어떤 사용자 문제를 몇 단계 줄이려 했나
- 설계: 데이터 계층과 렌더 계층을 어떻게 갈랐나 (`lib/engine/shared`의 transformToGraphData 재사용 등)
- 구현·배포·운영·개선: 배포 후 무엇을 보고 무엇을 바꿨나

## 2. 의사결정 = ADR 한 장

이미 ADR-0010(D3 force → Cytoscape.js 빅뱅 이관, 트리 뷰 React Flow 분리)을 남긴 흐름을 이어간다. 결정마다 왜 이걸 골랐고, 대안은 무엇이며, 트레이드오프는 무엇인지 적는다.

- 직접 만든 D3 엔진을 버리고 Cytoscape로 간 이유(유지 비용·제품 방향)
- 빅뱅 이관 vs 점진 이관에서 빅뱅을 택한 근거(미니맵·마퀴 일시 미구현 감수)

## 3. 숫자로 증명

Cytoscape는 Canvas, D3는 SVG로 렌더한다. 노드 수가 늘 때 렌더 비용 차이를 수치로 잡는다.

- 노드 100 / 1,000 / 5,000개에서 초기 렌더·인터랙션(zoom·pan) 시간을 DevTools Performance로 측정
- 이관·최적화 전후를 "X ms → Y ms"로 기록 → [[../FE/performance/core-web-vitals/웹-성능-측정]]

## 4. 트러블슈팅 일지

증상을 숨기지 말고 근본 원인을 고친 과정을 남긴다. 가설 → DevTools 확인 → 수정 순서로([[06-제대로된-코드-작성]] 원칙1). 블로그 글감으로도 좋다.

- Cytoscape는 Canvas·document가 필요해 클라이언트 전용이다. SSR 경계를 어떻게 처리했는지 → [[../FE/browser/렌더링-파이프라인]], [[../FE/react/하이드레이션]], [[../FE/debugging/크롬-devtools-디버깅]]

## 5. 부수효과·생명주기 체크

라이브러리 인스턴스와 이벤트를 등록한 곳에서 해제까지 책임진다([[06-제대로된-코드-작성]] 원칙2).

- Cytoscape 인스턴스 destroy, 리스너·레이아웃 정리, 언마운트 시 누수 없는지
- 클라이언트 전용 경계가 마운트/언마운트 반복에도 안전한지

## 6. 사용자 가치·운영 가설

1인·출시 전이라 사용자 지표가 없는 약점은, 본인 dogfooding에서 발견한 병목·문제정의로 메운다([[03-프로젝트-소유권]] MindGraph 항목). 배포 후에는 사용 흐름·트래픽을 기록해 운영 경험으로 전환한다.

## 7. 면접 방어로 연결

위 기록을 [[03-프로젝트-소유권]]의 MindGraph 재유도 질문(이관 이유, Canvas vs SVG 비용, 빅뱅 선택, 데이터/렌더 분리, SSR 경계, 사용자 가치) 답으로 옮겨 면접 노트를 채운다.

## 커밋·PR 올릴 때 자문

1. 이 처리가 근본 원인을 고치나, 증상을 숨기나
2. Cytoscape 인스턴스·리스너를 정리했나
3. 이 변경의 성능 영향을 측정했나(노드 수별 ms)
4. 이 결정은 ADR로 남길 만한가
5. 배포 후 무엇으로 효과를 확인할 건가
6. 6개월 뒤의 내가 이 코드와 결정을 이해하나
