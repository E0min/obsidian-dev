# 제대로 된 코드 작성 방향

> TL;DR: 동작하는 코드와 제대로 된 코드는 다르다. 증상을 숨기지 말고 근본 원인을 고치고, 부수효과는 격리하고 정리하고, 추측 대신 측정으로 최적화하고, 책임 경계로 쪼개고, 실패 경로를 먼저 설계한다. 본인 프로젝트 4개(Fit·chatGraph·xAB·MindGraph)의 실제 결정을 이 기준으로 다시 보면 어디가 정공법이고 어디가 회피였는지 보인다.

이 노트는 "무엇을 공부할까"([[05-채용시장-요구사항]])의 다음 단계인 "어떻게 짤까"를 다룬다. 학습 습관은 [[01-습관과-원칙]], AI와의 작업법은 [[04-ai-native-엔지니어]]에 있고, 여기서는 코드 자체의 품질 기준을 정리한다. 각 원칙에 본인 프로젝트의 실제 사례를 붙였다.

## 원칙 1. 증상이 아니라 근본 원인을 고친다

가장 중요한 원칙이다. 경고를 끄거나, 에러를 try/catch로 삼키거나, setTimeout으로 타이밍을 맞추는 처리는 대부분 증상을 숨긴 것이다. 그 순간 코드는 동작하지만 "왜 그래야 하는가"라는 모델을 우회하게 된다.

- **MindGraph 사례**: `reactStrictMode = false`로 zoom 이중 바인딩을 막은 건 증상 은폐다. 정공법은 effect 클린업으로 zoom 핸들러를 해제하는 것이다. StrictMode가 켜진 상태에서도 안전하게 짜면, "두 번 실행돼도 깨지지 않는 코드"라는 React의 멘탈 모델을 지키게 된다. 끄는 선택은 그 모델을 통째로 비활성화한다. [[../FE/react/strict-mode-이중호출]]
- **일반화**: 무언가를 끄거나 우회해서 문제가 사라졌다면, 먼저 "원래 왜 그게 켜져 있었나"를 물어야 한다.

## 원칙 2. 부수효과는 격리하고 반드시 정리한다

이벤트 리스너, 구독, 타이머, 외부 라이브러리 인스턴스는 등록한 자리에서 해제까지 책임진다. 등록만 하고 해제를 빠뜨리면 메모리 누수와 이중 바인딩이 쌓인다.

- **MindGraph 사례**: D3의 직접 DOM 조작을 `use client` 전용 레이어로 격리한 건 좋은 본능이다. 여기에 effect 클린업으로 생명주기까지 닫으면 원칙 1의 정공법과 맞물려 완성된다.
- **일반화**: `addEventListener`에는 `removeEventListener`를, 구독에는 해제를, `setInterval`에는 `clearInterval`을 짝으로 둔다. React라면 effect의 return으로.

## 원칙 3. 추측하지 말고 측정한 뒤 최적화한다

"빠를 것 같아서" 또는 "느릴 것 같아서" 내린 최적화는 대개 틀리거나 불필요하다. 측정으로 병목을 확인한 뒤에만 손댄다. 그리고 개선을 줄 수 같은 대리지표가 아니라 영향이 직접 드러나는 지표로 증명한다.

- **chatGraph 사례**: 843줄 훅을 355줄로 줄인 건 의미 있지만, 줄 수는 대리지표다. 리렌더 횟수, 번들 크기, 함수 복잡도가 분해의 효과를 직접 보여준다. [[../FE/react/리렌더링-트리거]]
- **MindGraph 사례**: 노드 100 / 1,000 / 5,000개에서 렌더·인터랙션 시간을 DevTools Performance로 재고, 가상화나 canvas 전환 같은 레버 하나를 적용해 "X ms → Y ms"로 증명한다. [[../FE/performance/core-web-vitals/웹-성능-측정]]

## 원칙 4. 책임 경계로 쪼갠다

한 함수나 모듈이 한 가지 이유로만 바뀌게 만든다. 경계가 분명하면 테스트하기 쉽고, 바꿀 때 영향 범위가 좁아진다.

- **chatGraph 사례**: 843줄 use-question-tree 훅을 책임 단위로 분해한 작업이 이 원칙의 모범이다. 쪼갤 때 기준은 줄 수가 아니라 "이 조각이 바뀌는 이유"여야 한다.
- **일반화**: 데이터 패칭, 상태 변환, 렌더링이 한 함수에 섞여 있으면 분리 후보다.

## 원칙 5. 실패 경로를 먼저 설계한다

해피 패스만 짜고 에러 처리를 나중에 붙이면, 실패는 사용자에게 그대로 노출된다. 무엇이 실패할 수 있는지와 그때 사용자에게 무엇을 보여줄지를 구현 전에 정한다.

- **Fit 사례**: setRemoteDescription 전 도착한 ICE candidate를 버퍼링한 건 실패 경로를 미리 설계한 예다. 더해서 "연결이 끝내 실패하면 사용자에게 무엇을 보여주는가"까지 정해야 완성이다. [[../CS/network/webrtc]]
- **xAB 사례**: optimistic 라우팅에서 mutation이 실패하면 temp ID 경로를 어떻게 롤백하는지가 곧 실패 경로 설계다.

## 원칙 6. 읽는 사람을 위해 쓴다

코드는 한 번 쓰고 여러 번 읽힌다. 이름이 의도를 드러내게 하고, 주석에는 "무엇"이 아니라 "왜"를 남기고, 변경은 되돌리기 쉬운 작은 단위로 올린다. 자기 결정을 ADR 한 장으로 남기는 습관이 설계 근육과 읽기 쉬운 코드를 동시에 만든다.

## 원칙 7. AI가 준 코드도 같은 기준을 통과시킨다

위 여섯 기준은 본인이 짠 코드든 AI가 짠 코드든 똑같이 적용된다. AI가 준 코드를 읽지 않고 머지하면 원칙 1~6을 전부 건너뛰는 것과 같다. 머지 전 한 줄씩 설명하고, "회의적인 리뷰어처럼 이 코드를 공격해봐"로 검증하는 작업법은 [[04-ai-native-엔지니어]]에 정리돼 있다.

## 적용 순서 (코드 짤 때 자문)

1. 이 처리가 근본 원인을 고치나, 증상을 숨기나
2. 내가 등록한 부수효과를 정리하고 있나
3. 이 최적화를 측정으로 확인했나
4. 이 함수가 한 가지 이유로만 바뀌나
5. 실패하면 사용자에게 무엇이 보이나
6. 6개월 뒤의 내가 이 코드를 읽고 이해하나
