# 채용 시장이 요구하는 것

> TL;DR: 스택은 입장권이라 변별력이 안 된다. 시장이 실제로 가르는 축은 ① 프레임워크 아래의 동작 원리, ② 성능을 수치로 다루는 측정 능력, ③ 상태·경계·데이터 흐름을 정하는 설계 능력, 그리고 ④ AI를 검증 루프 안에서 다루는 태도다. 12주 로드맵의 각 주가 이 네 축 중 어디를 채우는지 의식하며 공부한다.

## 큰 그림: 스택은 수렴했다

3~5년차 이상을 뽑는 공고에서 "React 쓸 줄 아세요"는 더 이상 질문이 아니다. React + TypeScript + Next.js, 서버 상태(React Query) / 클라이언트 상태(Zustand) 이원화, Emotion·Tailwind, Vite·Webpack, Jest·Playwright까지 상위 기업의 스택은 사실상 같은 조합으로 수렴했다. 그래서 시장이 정작 묻는 건 "불확실한 요구사항과 여러 직군 사이에서, 측정 가능한 품질을 유지하며 서비스를 끝까지 책임지고 개선할 수 있느냐"다.

이 노트는 그 요구를 공부 방향과 태도로 번역해 정리했다. 각 항목은 12주 로드맵([[00-로드맵-12주]])의 어느 주와 연결되는지 표시했다.

## 공부 방향 — 가로 확장은 멈추고 세로로 판다

스택을 하나 더 얹는 가로 확장(React 다음 Vue, Zustand 다음 Jotai)은 입장권을 한 장 더 받는 정도라 회수가 낮다. 아래 세 갈래로 깊이를 파야 변별력이 생긴다.

### 1) 동작 원리 — 프레임워크 아래의 변하지 않는 층

네이버 파이낸셜이 자격요건에 "브라우저와 서버 간 렌더링 상호작용, HTTP 메서드·상태코드·헤더, Reflow가 발생하는 이유와 방지책, 클로저, 제네릭, requestAnimationFrame이 필요한 상황에 대한 이해"를 명시하는 이유가 있다. 프레임워크는 3~4년마다 갈아끼워도 그 밑의 브라우저·네트워크·언어 런타임은 10년 단위로 거의 안 바뀌기 때문이다. 이 층을 말로 설명할 수 있으면 어떤 프레임워크로 옮겨도 디버깅 속도가 달라진다. AI가 코드를 생성해도 "왜 느린지" 진단하려면 사람이 이 모델을 머릿속에 갖고 있어야 하므로, AI가 못 메우는 자리가 여기다.

- 연결 노트: [[../FE/browser/렌더링-파이프라인]], [[../FE/javascript/클로저]], [[../FE/javascript/이벤트-루프]]
- 로드맵: 1~3주(JS 코어·브라우저), 6주(TypeScript)

### 2) 측정 능력 — '감'을 '수치'로 바꾼다

성능 측정·최적화 경험은 우대사항에서 점점 위로 올라온다. 네이버는 Chrome DevTools 성능 개선 경험을, 점핏은 웹 성능 측정·최적화 경험을 우대로 명시한다. 핵심은 Core Web Vitals(LCP·INP·CLS)를 이해하고, Lab 데이터(Lighthouse·DevTools)와 Field 데이터(web-vitals·CrUX)를 구분해 다루는 능력이다. 2024년 3월부터 INP가 FID를 대체했고, INP는 Field 데이터에서만 측정된다.

자기 템플릿 사이트 하나를 골라 "LCP 3.2초 → 1.8초로 줄였고 이미지 priority와 코드 분할이 레버였다"처럼 전후 숫자로 말할 수 있게 되면, 그게 포트폴리오 한 줄이자 면접 답변이 된다. 측정할 줄 아는 사람은 추측으로 싸우지 않는다.

- 연결 노트: [[../FE/performance/core-web-vitals/웹-성능-측정]], [[../FE/debugging/크롬-devtools-디버깅]]
- 로드맵: 7주(성능·디버깅)

### 3) 설계 — 컴포넌트 너머 상태·경계·데이터 흐름

서버 상태와 클라이언트 상태를 왜 분리하는지(캐싱·무효화·Stale-While-Revalidate), 공통 컴포넌트·디자인 시스템을 어떻게 경계 짓는지, 모듈을 어떻게 쪼개 의존성을 관리하는지. 당근은 NPM 패키지 출판 경험과 모듈 시스템 이해를 우대한다. 이 능력은 코드를 많이 짠다고 저절로 오지 않고, 자기 결정을 글로 설명해보면서 자란다. 작은 거라도 "왜 이 라이브러리를 택했는지" ADR 한 장씩 남기는 습관이 설계 근육을 만든다.

- 로드맵: 10주(프로젝트 소유권 — [[03-프로젝트-소유권]])

## 태도 — FE만이 아니라 모든 엔지니어의 축

기술이 비슷한 후보들 사이에서 최종을 가르는 건 일하는 자세다. 아래 다섯은 BE를 포함한 모든 개발 직군에 그대로 적용된다. 실천 방법은 [[01-습관과-원칙]]과 [[04-ai-native-엔지니어]]에 이미 정리돼 있으니, 여기서는 시장이 이걸 어떻게 문장으로 요구하는지만 매핑한다.

| 태도 | 시장이 쓰는 문장 | 실천 노트 |
|---|---|---|
| 문제를 코드보다 먼저 정의 | 토스 "주어진 개발을 해내기보다 주도적으로 문제를 발견·분석해 솔루션 제안" | [[04-ai-native-엔지니어]] 명세력 |
| 검증을 위임하지 않는다(AI 포함) | 예시 JD "팀 프로세스 안에서 품질을 유지하며 점진적으로 개선" | [[01-습관과-원칙]] 원칙 1, [[04-ai-native-엔지니어]] 검증력 |
| 자기 결정을 글로 남기고 설명 | 점핏 "자기 코드를 남에게 설명하는 게 어렵지 않은 분", 인프랩 "대화보다 문서를 작성·공유" | [[01-습관과-원칙]] 디버깅 기록 |
| 사용자·비즈니스 임팩트를 먼저 | 토스 "개발 이후에도 개선을 고민한 경험" | [[03-프로젝트-소유권]] |
| 지적 겸손 | 여러 공고 "자신이 항상 틀릴 수 있다는 전제로 커뮤니케이션" | [[01-습관과-원칙]] |

AI를 다루는 태도가 지금의 분기점이다. Stack Overflow 2024 기준 개발자 76%가 AI 코딩 도구를 쓰므로 "AI를 쓰느냐"는 이미 변별점이 아니다. METR 연구에서 검증 없이 받아쓰면 숙련 개발자도 오히려 느려지는 사례가 관찰됐다. 그래서 생성·리팩토링·테스트 초안은 위임하되, 맞는지 판단하고 팀 컨벤션에 녹이고 책임지는 일은 사람 몫으로 남긴다. 자세한 작업법은 [[04-ai-native-엔지니어]].

## 한 줄 정리

빠르게 코드 짜는 능력의 값은 AI 때문에 내려가고, 무엇을 만들지 정의하고·왜 그런지 설명하고·맞는지 검증하는 능력의 값은 올라간다. 그러니 공부는 프레임워크 위가 아니라 그 아래(동작 원리)와 그 위(측정·설계)로 파고, 태도는 "생성은 위임하되 판단과 책임은 끝까지 쥐는" 쪽으로 가져간다.

## 자가 점검 (분기마다)

- [ ] 동작 원리: 렌더링 파이프라인과 Reflow 조건, 이벤트 루프, HTTP 캐시를 말로 설명할 수 있나
- [ ] 측정: 내 프로젝트 하나의 Core Web Vitals를 전후 숫자로 말할 수 있나
- [ ] 설계: 최근 기술 선택 하나에 대해 ADR을 남겼나
- [ ] 태도: 이번 분기에 AI가 준 코드를 검증 없이 머지한 적이 있나 (있으면 [[01-습관과-원칙]] 복귀)
